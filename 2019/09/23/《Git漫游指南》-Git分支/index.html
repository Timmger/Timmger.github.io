《Git漫游指南》- Git分支

发表于 2019-09-23 | 分类于 Git | | 阅读次数:
字数统计: 2k | 阅读时长 ≈ 7

1. 杀手级特性

使用分支可以把工作从开发主线上分离开来，以免影响开发主线。Git 处理分支的方式可谓是难以置信的轻量，创建新分支这一操作几乎能在瞬间完成，并且在不同分支之间的切换操作也是一样便捷。

1.1 git的数据保存

在git commit 进行提交操作时，Git仓库会有五个对象：三个 blob 对象（保存着文件快照）、一个树对象（记录着目录结构和 blob 对象索引）以及一个提交对象（包含着指向前述树对象的指针和所有提交信息）。

图片来自《Pro Git》

若修改后再提交，会产生指向上次提交对象的指针

Git 的分支，其实本质上仅仅是指向提交对象的可变指针
master 分支, 它会在每次的提交操作中自动向前移动。

1.2 分支创建

git branch testing: 创建一个testing分支

在当前提交对象上创建

HEAD 为当前分支的别名
使用git log查看当前分支所指对象

1.3 分支切换

git checkout testing

若再次修改提交，HEAD 分支随着提交操作自动向前移动

切换回master分支并作修改

项目产生分叉

查看分叉历史

Git 的分支实质上仅是包含所指对象校验和（长度为 40 的 SHA-1 值字符串）的文件，所以它的创建和销毁都异常高效。

2. 分支的新建与合并

紧急任务修复

  1. 切换到你的线上分支（production branch）。

  2. 为这个紧急任务新建一个分支，并在其中修复它。

  3. 在测试通过之后，切换回线上分支，然后合并这个修补分支，最后将改动推送到线上分支。

  4. 切换回你最初工作的分支上，继续工作。

2.1 新建分支

git checkout -b iss53: 新建iss53并同时切换

修改并提交

此时遇到紧急问题，切换回master并新建hotfix

使用git merge合并

$ git checkout master
$ git merge hotfix

此时master分支快进到hotfix

切换回之前工作

先删除hotfix

git branch -d hotfix

git checkout iss53

2.2 合并

在iss53完成后，合并到master分支
master 分支所在提交并不是 iss53 分支所在提交的直接祖先
Git 会做一个简单的三方合并。


此时删除iss53

git branch -d iss53

2.3 遇到冲突时的合并

两个不同的分支中，对同一个文件的同一个部分进行了不同的修改，Git 就没法干净的合并它们。

使用 git status 命令来查看未合并（unmerged）状态的文件。
解决了所有文件里的冲突之后，对每个文件使用 git add 命令来将其标记为冲突已解决。

3. 分支管理

git branch可显示分支列表

git branch -v查看每一个分支的最后一次提交

–merged (已合并) 与 –no-merged（未合并）

未合并的分支不可用git branch -d删除

4. 分支开发工作流

4.1 长期分支

通常把他们想象成流水线（work silos），经过测试考验的提交会被遴选到更加稳定的流水线上去。

4.2 特性分支

特性分支是一种短期分支，它被用来实现单一特性或其相关工作。使你快速并且完整地进行上下文切换（context-switch）

5. 远程分支

远程跟踪分支是远程分支状态的引用，它们以 (remote)/(branch) 形式命名。
克隆之后的服务器与本地仓库

本地与远程的工作可以分叉，只要不与 origin 服务器连接，你的 origin/master 指针就不会移动。

此时从远程服务器增量更新, origin/master 指针指向新的、更新后的位置。

git fetch origin

添加另一个远程仓库，并追踪分支

5.1 推送

git push origin serverfix: 在origin远程仓库上分享serverfix分支
git push origin serverfix:coolbranch: 将本地的 serverfix 分支推送到远程仓库上的 coolbranch 分支

其他协作者从服务器上抓取数据时,本地生成一个远程分支 origin/serverfix, 如果想要在自己的 serverfix 分支上工作，可以将其建立在远程跟踪分支之上

5.2 跟踪分支

当克隆一个仓库时，会自动地创建一个跟踪 origin/master 的 master 分支。
可以设置其他的跟踪分支

与 git checkout -b serverfix origin/serverfix 等价

本地分支与远程分支设置为不同名字

设置已有的本地分支跟踪刚拉取的远程分支

查看设置的所有跟踪分支

git branch -vv

6. 变基

整合来自不同分支的修改主要有两种方法：merge 以及 rebase

6.1 变基的基本操作

整合分支最容易的方法是 merge ，进行三方合并，合并的结果是生成一个新的快照。

还有一种方法为变基：提取在 C4 中引入的补丁和修改，然后在 C3 的基础上应用一次

回到 master 分支，进行一次快进合并

6.2 更多

对于特性分支再分出特性分支

将 client 中的修改合并到主分支并发布，但暂时并不想合并 server 中的修改
选中在 client 分支里但不在 server 分支里的修改（即 C8 和 C9），将它们在 master 分支上重放

$ git rebase –onto master server client

快进合并 master 分支

git checkout master
git merge client

此时变基server分支

git rebase master server

快进合并并删除

git checkout master
git merge server
git branch -d client
git branch -d server

6.3 变基的风险

黄金准则

不要对在你的仓库外有副本的分支执行变基。

  1. 克隆一个仓库，然后在它的基础上进行了一些开发

  2. 抓取别人的提交，合并到自己的开发分支

  3. 有人推送了经过变基的提交，你从服务器抓取更新，会发现多出来一些新的提交

  4. 执行 git pull 命令, 相同的内容又合并了一次

C4 和 C6，通过变基丢弃的提交

6.4 用变基解决变基

在步骤3时，不进行合并，而执行

git rebase teamone/master

另一种简单的方法

git pull –rebase

因此只在从未推送至共用仓库的提交上执行变基命令，就不会有事。
